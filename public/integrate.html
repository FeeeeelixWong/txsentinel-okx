INTEGRATION RUNBOOK

Four steps in the agent loop

  1. 01
    Construct the proposed action

    Include chain, operation, destination, amount, user policy, and simulation evidence.

  2. 02
    POST before signing

    Call TxSentinel after transaction construction but before wallet authorization.

  3. 03
    Branch on the decision

    ALLOW may continue, HOLD needs human review, and DENY must stop.

  4. 04
    Persist the receipt

    Store the normalized request and receipt hash, or anchor it on X Layer.

MINIMUM REQUEST

Copy a working request

curl -X POST https://txsentinel-okx.vercel.app/api/check \
  -H "Content-Type: application/json" \
  -d '{
    "chain": "xlayer",
    "operation": "transfer",
    "to": "0x4a6aae28b27681856ae824af82fea87896ecc3ed",
    "amountUsd": 25,
    "policy": {
      "maxSpendUsd": 100,
      "denyUnlimitedApprovals": true,
      "requireSimulation": true
    },
    "simulation": {
      "status": "succeeded",
      "estimatedFeeUsd": 0.01
    }
  }'

INTEGRATION BOUNDARY

Keep authority where it belongs

YOUR AGENT OWNS

Transaction construction, RPC access, private configuration, and downstream execution.

TXSENTINEL OWNS

Schema validation, deterministic policy logic, reason codes, and stable receipt hashes.

TXSENTINEL NEVER GETS

Private keys, seed phrases, wallet sessions, signing authority, or broadcast permissions.
